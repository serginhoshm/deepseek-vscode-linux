## What is Continue.dev?

Continue is an open-source VS Code (and JetBrains) extension that integrates LLMs directly into the editor. Its three main features are:

| Feature | How to trigger | Description |
| :--- | :--- | :--- |
| **Chat** | `Ctrl+L` | Conversational assistant with codebase context |
| **Autocomplete** | `Tab` | Inline ghost-text completion as you type |
| **Edit** | `Ctrl+I` | Select code, describe a change, apply it in-place |

The `@codebase` context selector uses an embeddings model to do semantic search over your repository, letting the chat answer questions about code it has never seen in the current conversation.

---

## Installation

Search for **Continue** in the VS Code Extensions Marketplace, or install it from the command line:

```bash
code --install-extension continue.continue
```

Verify the installed version:

```bash
code --list-extensions --show-versions | grep continue
# Expected: continue.continue@2.0.0 (or newer)
```

## Model Stack

Continue 2.0 separates models by **role** (see [Model Roles](#model-roles)). This setup uses four Ollama models:

| Role | Model | Why this model |
| :--- | :--- | :--- |
| Chat / Edit / Apply | `llama3.1:8b` | General-purpose; clean output, no reasoning noise |
| Chat / Edit / Apply | `qwen2.5-coder:7b` | Code-focused; strong at refactoring and generation |
| Autocomplete | `qwen2.5-coder:1.5b` | Fast enough for inline completion; fill-in-the-middle trained |
| Embeddings | `nomic-embed-text:latest` | Required for `@codebase` semantic search |

### Why not deepseek-r1?

DeepSeek R1 wraps all reasoning inside `<think>...</think>` XML tags before answering. Continue renders model output as plain text — it does not strip or collapse those tags — so the entire chain-of-thought appears verbatim in the chat panel, making responses unreadable for normal use.

Use DeepSeek R1 directly via `ollama run deepseek-r1:7b` in the terminal, or via **Cline / Roo Code** (which handles streaming output differently).

### Pulling the models

```bash
ollama pull llama3.1:8b
ollama pull qwen2.5-coder:7b
ollama pull qwen2.5-coder:1.5b
ollama pull nomic-embed-text:latest
```

Approximate disk usage: ~11 GB total.

## Configuration File

### Location

Continue 2.0 reads its configuration from:

```
~/.continue/config.yaml
```

Older versions used `~/.continue/config.json`. Continue 2.0 still reads `config.json` as a fallback, but **`config.yaml` takes precedence** when both files exist. The extension will also prompt to migrate `config.json` to `config.yaml` automatically on first launch.

### Required header fields

Continue 2.0 requires three header fields at the top of `config.yaml`. Without them, the extension silently fails to parse the file and shows **"No models configured"**:

| Field | Required | Value |
| :--- | :--- | :--- |
| `name` | Yes | Any string label for this config |
| `version` | Yes | Semantic version string, e.g. `0.0.1` |
| `schema` | Yes | Must be `v1` — tells the parser which format to use |

> **The `schema: v1` field is the most common cause of "No models configured" in Continue 2.0.** Configs copied from older docs or generated by older tools will be missing it.

---

## Model Roles

In Continue 2.0, each model entry declares its `roles` — a list of editor features the model is allowed to power. A model that is not assigned a role for a given feature will not appear in that feature's selector.

| Role | Feature |
| :--- | :--- |
| `chat` | Chat panel (`Ctrl+L`) |
| `edit` | Inline edit (`Ctrl+I`) |
| `apply` | Apply suggested code changes |
| `autocomplete` | Tab autocomplete |
| `embed` | `@codebase` semantic search |
| `rerank` | Re-ranking search results (optional) |
| `summarize` | Summarization tasks |
| `subagent` | Agent mode sub-tasks |

The `tabAutocompleteModel` top-level key (used in pre-2.0 configs) is **no longer needed** — set `roles: [autocomplete]` on the model entry instead.

## Full config.yaml Reference

This is the complete working configuration written by `setup.sh`:

```yaml
name: Local Ollama Config
version: 0.0.1
schema: v1

models:
  - name: Llama 3.1 8B
    provider: ollama
    model: llama3.1:8b
    apiBase: http://localhost:11434
    roles:
      - chat
      - edit
      - apply

  - name: Qwen 2.5 Coder 7B
    provider: ollama
    model: qwen2.5-coder:7b
    apiBase: http://localhost:11434
    roles:
      - chat
      - edit
      - apply

  - name: Qwen 2.5 Coder 1.5B
    provider: ollama
    model: qwen2.5-coder:1.5b
    apiBase: http://localhost:11434
    roles:
      - autocomplete

  - name: Nomic Embed
    provider: ollama
    model: nomic-embed-text:latest
    apiBase: http://localhost:11434
    roles:
      - embed
```

### Field reference

| Field | Description |
| :--- | :--- |
| `name` | Display name shown in the Continue model selector |
| `provider` | LLM provider — `ollama` for local models |
| `model` | Exact model tag as returned by `ollama list` |
| `apiBase` | Ollama endpoint; defaults to `http://localhost:11434` if omitted, but explicit is safer |
| `roles` | List of Continue features this model is allowed to power |

## Using @codebase

`@codebase` is the Continue feature that lets the chat model answer questions about your entire project, not just what is visible in the current file. It requires the embeddings model (`nomic-embed-text`) to be configured and indexed.

### How it works

1. When VS Code opens a workspace, Continue starts indexing it in the background using `nomic-embed-text`. This process chunks all source files, generates embedding vectors, and stores them in `~/.continue/index/`.
2. When you type `@codebase` in the chat, Continue runs a semantic search over the index to find the most relevant code chunks and injects them as context before sending your message to the chat model.
3. The chat model then answers with awareness of your full project, not just the currently open file.

### Activating it

In the Continue chat panel, type `@codebase` followed by your question:

```
@codebase Where is the authentication middleware defined?
@codebase What does the setup.sh logging system do?
```

The `@codebase` tag must be **explicit** — it does not activate automatically. Continue will not inject repository context unless you use it.

### Watching indexing progress

Indexing runs silently. To see its progress:

```
View → Output → select "Continue" from the dropdown
```

On first load of a large project, indexing can take several minutes. The `@codebase` results will be incomplete until indexing finishes.

### Why results may be poor

- **Context window too small** — the chat model receives many chunks from the index, but if its context window is small, older chunks get dropped. See [Ollama Context Window](#ollama-context-window).
- **Index is stale** — if you add many new files, force a re-index: Continue panel → gear icon → **Re-index**.
- **Model quality** — `nomic-embed-text` at 137M parameters is lightweight. For large monorepos, results may miss relevant files. Consider a larger embeddings model if accuracy matters.

## Ollama Context Window

When reading multiple files simultaneously — as happens with `@codebase` or long chat sessions — the model's context window is the binding constraint. If the context fills up, Ollama silently drops the oldest content from the conversation, causing the model to "forget" earlier files or instructions.

### Default context sizes

Ollama sets a default `num_ctx` (context window) of **2048 tokens** for most models unless overridden. This is very small for code tasks. The models in this setup support much larger windows:

| Model | Maximum context | Recommended `num_ctx` |
| :--- | :--- | :--- |
| `llama3.1:8b` | 131 072 tokens | 8 192 – 32 768 |
| `qwen2.5-coder:7b` | 32 768 tokens | 8 192 – 32 768 |
| `qwen2.5-coder:1.5b` | 32 768 tokens | 4 096 (autocomplete only) |

Larger `num_ctx` consumes proportionally more VRAM/RAM. On a 16 GB RAM machine, 8 192–16 384 is a safe range for the 7–8B models.

### Setting context length in config.yaml

The cleanest way is to declare `contextLength` per model in Continue's config, which passes `num_ctx` to Ollama on every request:

```yaml
name: Local Ollama Config
version: 0.0.1
schema: v1

models:
  - name: Llama 3.1 8B
    provider: ollama
    model: llama3.1:8b
    apiBase: http://localhost:11434
    contextLength: 16384
    roles:
      - chat
      - edit
      - apply

  - name: Qwen 2.5 Coder 7B
    provider: ollama
    model: qwen2.5-coder:7b
    apiBase: http://localhost:11434
    contextLength: 16384
    roles:
      - chat
      - edit
      - apply

  - name: Qwen 2.5 Coder 1.5B
    provider: ollama
    model: qwen2.5-coder:1.5b
    apiBase: http://localhost:11434
    contextLength: 4096
    roles:
      - autocomplete

  - name: Nomic Embed
    provider: ollama
    model: nomic-embed-text:latest
    apiBase: http://localhost:11434
    roles:
      - embed
```

### Setting context length via a custom Modelfile

Alternatively, create a persistent Ollama model variant with a baked-in context size:

```bash
cat > /tmp/Modelfile-qwen-large << 'EOF'
FROM qwen2.5-coder:7b
PARAMETER num_ctx 16384
EOF
ollama create qwen2.5-coder:7b-ctx16k -f /tmp/Modelfile-qwen-large
```

Then reference `qwen2.5-coder:7b-ctx16k` in `config.yaml`. This avoids relying on Continue to pass the parameter correctly.

### Checking current context size at runtime

```bash
curl -s http://localhost:11434/api/show \
  -d '{"name":"qwen2.5-coder:7b"}' \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('model_info',{}).get('qwen2.context_length') or d.get('parameters',''))"
```

## Verifying the Setup

### 1. Confirm Ollama is running and all models are present

```bash
curl -s http://localhost:11434/api/tags | python3 -m json.tool | grep '"name"'
```

Expected output includes all four model names.

### 2. Send a test prompt from the terminal

```bash
curl -s http://localhost:11434/api/generate \
  -d '{"model":"llama3.1:8b","prompt":"Reply with one word: hello","stream":false}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['response'])"
```

### 3. Reload VS Code

After editing `config.yaml`, reload the window for Continue to pick up the changes:

```
Ctrl+Shift+P → Developer: Reload Window
```

### 4. Check the Continue panel

Open the Continue panel (`Ctrl+L`). The model selector at the top should list **Llama 3.1 8B** and **Qwen 2.5 Coder 7B**. Autocomplete activates automatically when you type in any editor file.

## Troubleshooting

### "No models configured"

This is almost always caused by a missing or wrong header field in `config.yaml`.

**Check:**

```bash
head -5 ~/.continue/config.yaml
```

Expected output:
```yaml
name: Local Ollama Config
version: 0.0.1
schema: v1
```

If `schema: v1` or `name` is missing, add them and reload VS Code.

---

### Models appear in the selector but return no response

Ollama is not running or not reachable.

```bash
curl http://localhost:11434/
# Should return: Ollama is running
```

If it fails, start the service:

```bash
sudo systemctl start ollama
# or, without systemd:
ollama serve &
```

---

### Autocomplete is not triggering

1. Confirm `qwen2.5-coder:1.5b` is pulled: `ollama list | grep qwen`
2. Confirm the model has `roles: [autocomplete]` in `config.yaml`
3. Check that Continue's autocomplete toggle is enabled: gear icon in the Continue panel → Autocomplete → enabled

---

### `@codebase` returns no results

The embeddings model needs to index the workspace first. This happens automatically in the background when Continue starts. Progress is visible in the Continue output panel (`View → Output → Continue`).

If the index never builds, check that `nomic-embed-text:latest` is present in `ollama list` and has `roles: [embed]` in `config.yaml`.

---

### config.yaml vs config.json — which one wins?

In Continue 2.0, `config.yaml` always wins when both files exist. The extension will show a notification offering to delete the old `config.json`. You can also delete it manually:

```bash
rm ~/.continue/config.json
```

---

### How to open config.yaml from VS Code

Click the **gear icon** at the bottom-right corner of the Continue panel, then select **Open Config**. This opens `~/.continue/config.yaml` directly in the editor.

## Alternatives to Continue

If the Continue experience is slow, lacks context, or causes VS Code to hang on lower-end hardware, these extensions are worth trying:

### Twinny

A fully local, privacy-first alternative to Continue. Works with Ollama out of the box with minimal configuration — no YAML schema requirements, no indexing pipeline to manage. Good default choice when you want autocomplete and chat without the overhead.

- VS Code marketplace: search **Twinny**
- Configure: Settings → Twinny → set Ollama endpoint to `http://localhost:11434` and pick a model

### Llama Coder

Focused exclusively on tab autocomplete. If inline completion is your primary need and Continue's autocomplete feels slow or unreliable (especially on CPU-only setups), Llama Coder has less overhead and tends to produce faster ghost-text suggestions.

- VS Code marketplace: search **Llama Coder**
- Works directly with Ollama; set the FIM model to `qwen2.5-coder:1.5b`

### When to stick with Continue

Continue remains the best option when you need `@codebase` semantic search, the Edit (`Ctrl+I`) workflow, or plan to use cloud providers alongside local models. The complexity of its configuration pays off at that feature level.

## Cloud API Alternative

To use the DeepSeek cloud API instead of (or alongside) local Ollama models, generate an API key at [platform.deepseek.com](https://platform.deepseek.com/) and add entries to `config.yaml`:

```yaml
name: Mixed Config
version: 0.0.1
schema: v1

models:
  - name: DeepSeek V3 (Cloud)
    provider: deepseek
    model: deepseek-chat
    apiKey: YOUR_API_KEY_HERE
    roles:
      - chat
      - edit
      - apply

  - name: DeepSeek R1 (Cloud Reasoning)
    provider: deepseek
    model: deepseek-reasoner
    apiKey: YOUR_API_KEY_HERE
    roles:
      - chat

  - name: Qwen 2.5 Coder 1.5B (local autocomplete)
    provider: ollama
    model: qwen2.5-coder:1.5b
    apiBase: http://localhost:11434
    roles:
      - autocomplete

  - name: Nomic Embed (local)
    provider: ollama
    model: nomic-embed-text:latest
    apiBase: http://localhost:11434
    roles:
      - embed
```

> Cloud R1 is safe to use for chat in Continue because the cloud API strips `<think>` tags from the response before returning it. Only the local Ollama version of R1 exposes them.
